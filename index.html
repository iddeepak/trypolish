📚 Software Design Principles – Quick Examples

A curated list of key OOP/design principles with short, real-world code examples.


🅰️ A – Abstractions & Class Design

  • Abstractions Live Longer Than Details: Use List over ArrayList.
  • Add More Classes: Split UserService into UserRepo, EmailService, etc.

🅲 C – Cohesion, Coupling & Crash

  • Constantine's Law: High cohesion + low coupling = stability.
  • Crash Early: Throw exceptions early, like null checks in constructors.
  • Curly’s Law: Do one thing. Follow SRP.

🅳 D – Dependency & Duplication

  • DIP: Depend on interfaces, not concrete classes.
  • Direct Mapping: Map UserDTO ↔ UserEntity.
  • DRY: Don’t repeat yourself — use helpers.
  • Law of Demeter: Avoid user.getProfile().getCity(); use user.getCity().

🅴 E – Explicitness & Encapsulation

  • Easy to Use, Hard to Misuse: Use builders or fluent APIs.
  • Explicit > Implicit: Prefer enum Status over strings.

🅵 F – Failures & Fallacies

  • Fail Fast: Validate early.
  • Fallacies of Distributed Computing: Network is not reliable. Plan for it.

🅶 G – Generalization & Growth

  • Gall’s Law: Build simple first.
  • Postel’s Law: Be lenient in input, strict in output.
  • Generalization Principle: Use generic types.

🅷 H – High Cohesion

  • High Cohesion: A class like ReportService should only generate reports.

🅸 I – Interfaces & Information

  • Information Expert: Let object with data make decisions using that data.
  • Encapsulation: Keep fields private and expose behavior.
  • ISP: Split large interfaces into smaller focused ones.
  • IAP: Avoid exposing mutable state directly.

🅺 K – Keep it Simple

  • KISS: Prefer a simple for loop over complex streams.

🅻 L – Laws & Loops

  • LoD: Talk only to direct collaborators.
  • LLA: Abstractions leak, like ORM and SQL joins.
  • LSP: Don’t break logic when using subclass.
  • Low Coupling: Favor interfaces or events.

🅼 M – Memory & Modeling

  • Miller’s Law: Limit data shown at once.
  • Minimize Coupling: Use messaging patterns.
  • Model Principle: Reflect real-world entities in code.
  • MIMC: Avoid overengineering early.
  • Murphy’s Law: Handle exceptions & nulls.

🅾️ O – OOP Basics

  • One Responsibility: One reason to change per class.
  • OCP: Open for extension, closed for modification.

🅿️ P – Principles for People

  • Pit of Success: Make the right path the easiest.
  • PLA: Avoid surprising behaviors.
  • LoD: Don't chain too much.
  • Least Privilege: Grant minimum access required.
  • PLS: Software should do what user expects.
  • Separate Understandability: Each module should be clear alone.

🆁 R – Rules to Code By

  • Explicitness: Be clear over clever.
  • Repair Fast: Fail early and clearly.
  • Powerful Patterns: Use Strategy, Builder, etc.

🆂 S – Separation & Structure

  • SoC: Separate UI, service, DB, etc.
  • SLA: Each function on one level of abstraction.
  • SRP: One job per class.
  • SSOT: Don’t duplicate sources of truth.

🆃 T – Tell Don’t Ask

  • TdA: Prefer asking user.getDiscount() than inspecting fields outside.

🆄 U – Uniform Design

  • Uniformity: Stick to consistent naming, structure, etc.

🆈 Y – YAGNI

  • You Ain’t Gonna Need It: Don’t build what isn’t needed now.

🆉 Z – Zero One Infinity

  • ZOI: Either support none, one, or infinite — not arbitrary limits.
